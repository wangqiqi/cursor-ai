# 文档预审清单（吸收自 SpaceZephyr/pm-skills · MIT）

> 出处 → **`docs/library-index.md`**（§ pm-skills）· 本文件为操作 SSOT。  
> **用这个**：PRD / 原型 / 功能说明过会前预审。**不是那个**：代码 PR 回顾（**review** 主清单）· 上线 7 维（**delivery**）。

## 严格度（先定档）

| 档位 | 场景 | 口径 |
|------|------|------|
| 快速过筛 | 需求仍发散 | 仅阻断项 + 结构性问题；每视角 ≤2 条 |
| 标准评审（默认） | 进评审会前预演 | 六视角全量 + 三级风险 |
| 上线前终审 | 提测/明天上线 | 合规/运营/测试提权：相关项一律可升阻断 |

材料仅一段话或几张截图 → 输出**材料补充建议**，不硬评；建议先走 **plan** §协作文档。

## 六视角（泛化名 · 禁止专家人名）

| 视角 | 关注 | 典型问题 |
|------|------|----------|
| **产品完整性** | 目标、范围、用户价值、优先级 | 目标不可量化 · 范围蔓延 · MVP 不清 |
| **研发可行性** | 数据结构、接口、性能、工作量 | 字段未定义 · 并发未说明 · 依赖未确认 |
| **测试可验收性** | 验收标准、异常路径、边界 | 验收不可执行 · 异常未覆盖 |
| **设计交互** | 一致性、空错态、反馈、无障碍 | 空状态缺失 · 操作反馈不明 |
| **运营与数据** | 埋点、灰度、回滚、指标 | 漏斗无埋点 · 无回滚方案 |
| **合规与安全** | 隐私、数据合规、UGC、协议 | 收集信息未告知 · 无审核机制 |

用户只要单视角（如「研发能不能做」）→ 只输出该视角 + 一句「需要完整预审可说一声」。

## 意见质量红线

每条意见须同时满足，缺一删除：

1. **指得到位置** — 章节/段落/界面元素（禁止「整体不够清晰」）
2. **说得清后果** — 不修会在哪炸（返工/无法验收/事故/合规）
3. **给得出动作** — 具体补什么/改成什么（禁止「建议优化」）

## 风险分级

| 级 | 定义 |
|----|------|
| **阻断** | 核心流程不通 · 安全合规 · 资金风险 · 技术不可行且无替代 · 验收完全缺失 |
| **重要** | 异常未覆盖（非资金）· 体验明显问题 · 埋点缺失 · 边界未定义 |
| **建议** | 文案/细节 · 可下版补充 · 文档格式 |

多视角重复 → 合并并提权；矛盾 → 列双方观点 + **`Decision needed`**，不替用户拍板。

## 总体判定

| 条件 | 判定 |
|------|------|
| 0 阻断 + 0 重要 | ✅ 通过 |
| 0 阻断 + 有重要 | ⚠️ 有条件通过（重要项提测前修） |
| 有阻断 | ❌ 不通过 |

## 输出结构（摘要）

```markdown
# 文档预审结论

**判定**：✅ / ⚠️ / ❌ — 一句话理由

## 阻断 / 重要 / 建议（表格：问题 · 视角 · 位置 · 修改建议）

## 各视角意见（2–5 条/视角）

## 复评清单（修改项 · 对应编号 · 复评标准）
```

结构性混乱（非细节问题）→ 最多 3 条核心结构问题 + 建议回到 **plan** §协作文档重写。

## 与代码 review 分工

| 材料 | 入口 |
|------|------|
| PRD / 原型 / 需求 doc | 本清单 |
| 代码 diff / PR | **review** Standards/Spec 双轴 |
| 实现后上线 | **delivery** |
